## Findings considered and rejected (this session)

- **RLS "window of exposure" on the users SELECT policy**: rejected in first
  audit. Final migration state is restrictive; `users_public` view serves
  author names with least privilege.
- **"Saved posts feature is missing from UI"**: rejected in first audit.
  Feature IS wired — the defect was the toggle mutation (fixed in plan 002).
- **Supabase anon key exposed in client bundle**: rejected — intended Supabase
  architecture; authorization enforced by RLS.
- **`as any` casts in production code**: first audit found 0; current audit found
  `any` in service parameters/returns (plan 011), not `as any` casts.
- **Pages calling services directly (DEBT-02)**: noted but not planned — larger
  refactor (L effort) with MED risk; individual instances are being addressed
  incidentally by other plans.
- **Hard-coded .limit(50) with no "load more"**: noted but not planned — worth
  a pagination plan if lists are expected to exceed 50 in practice; current
  user base is small.
- **No ESLint/Prettier**: noted but not planned — low urgency given strict TS,
  but adds no dead-code/style enforcement.
- **useDarkMode doesn't respect prefers-color-scheme on first load**: noted as
  minor a11y gap; reads only localStorage. Worth fixing but LOW impact.
- **usePresence beforeunload unreliability**: noted; 60s keepalive mitigates but
  doesn't fully solve. LOW impact for current user base.

## Notable findings NOT turned into plans (lower leverage / larger decisions)

- **No rate limiting on memberRequestService.createRequest (Supabase insert)**:
  the `/api/admin/notify-request` Express endpoint has rate limiting, but the
  Supabase INSERT path doesn't. Worth adding if spam becomes an issue; RLS
  "Anyone create requests" is by design (pre-auth). A Supabase Edge Function
  or rate-limit trigger could address this.
- **8+ pages with zero test coverage**: Messages, Notifications, Postos,
  PostoDetails, AdminHub, AdminMembers, AdminModeration, CarreiraPromocao,
  Aposentadoria, ForgotPassword, RegisterRequest. Large effort; prioritize
  pages with business logic (AdminModeration, RegisterRequest) if coverage
  expansion is desired.
- **Hardcoded "Trending Topics" in Feed sidebar**: should be dynamic based on
  actual post data. Medium effort for a server-side function.
- **New-member onboarding flow**: after approval, members land on /feed with
  limited guidance. A lightweight onboarding checklist would reduce cold-start
  friction for the 65-85 persona.
