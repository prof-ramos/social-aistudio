## Findings considered and rejected

So these are not re-audited next run:

- **RLS "window of exposure" on the `users` SELECT policy** (subagent SECURITY-01):
  rejected. The final migration state is `USING (auth.uid() = id)` (restrictive) and
  author names are served via the least-privilege `users_public` view. The
  intermediate permissive policy only ever existed transiently during forward
  migration ordering — by-design, not a live vulnerability.
- **"Saved posts feature is built but never surfaced in the UI"** (subagent
  DIRECTION-01): rejected as false. The bookmark IS wired —
  `src/pages/Feed.tsx:209` passes `onToggleSaved`, `src/components/feed/PostCard.tsx:79-83`
  renders the bookmark button. The real defect is that the toggle doesn't re-render
  (covered by plan 002), not that the feature is missing.
- **Supabase anon key exposed in the client bundle** (subagent SECURITY-06):
  rejected as a finding. Shipping the anon key client-side is the intended Supabase
  architecture; authorization is enforced by RLS. (A backend-for-frontend proxy is a
  possible future direction, not a defect.)
- **"76 `as any` casts in src"** (subagent DX-01 sub-claim): rejected — inaccurate.
  `grep -rn "as any" src/` (excluding tests) returns 0. Strict mode is on and clean.
  A couple of `useState<any>` exist (e.g. `usePostDetails.ts`) — minor, not planned.

## Notable findings NOT turned into plans (lower leverage / larger decisions)

Recorded for the maintainer; pick up later if desired:

- **Pages calling services directly** (DEBT-02) — `Feed.tsx`, `PostDetails.tsx`,
  `Messages.tsx` call services directly, against the CLAUDE.md "pages use hooks" rule.
  Plan 002 incidentally removes one such call (`userService.toggleSavedPost` in Feed).
  A broader consolidation is a separate M refactor.
- **Hard-coded `.limit(50)` with no "load more" on chats, notifications, admin lists,
  postos** (PERF-03/09) — silent truncation. Worth a pagination plan if any list is
  expected to exceed 50 in practice.
- **Eager-loaded heavy deps on the login route** (PERF-06) — `react-joyride` (Tour)
  and `motion` are imported at module level and ship on first paint; the 553KB chunk
  flagged in `READY-TO.md` is related. A code-splitting plan would help the elderly /
  slow-device audience.
- **In-memory rate limiter ineffective on serverless** (SECURITY-03) — plan 003
  de-duplicates it but explicitly leaves the durability gap; a Redis/KV-backed limiter
  is the real fix for the production (Vercel) path.
- **No ESLint/Prettier; `lint` is `tsc` only** (DX-01) — low urgency given strict TS,
  but adds no dead-code/style enforcement.
- **No ADRs / domain glossary** (DIRECTION-03, flagged in `READY-TO.md`) — onboarding
  friction; a docs task, not code.
