## Anti-Patterns Verdict

**LLM Assessment**: The interface reads as intentionally designed, not AI-generated. The navy/ice palette is systematic and restrained, the Cormorant Garamond + Source Sans 3 pairing is distinctive, and the sharp-cornered (0px radius) system is a clear brand commitment. However, three specific patterns betray AI influence:

1. The `border-l-4` on PostoHighlightCard is a textbook side-tab accent — the single most recognizable AI-UI tell.
2. The chat bubbles in Messages use `rounded-t-2xl rounded-l-2xl rounded-br-sm`, the standard LLM chat bubble pattern.
3. The Navbar's `border-b-2` active indicator on items with `rounded-none` creates an inconsistent shape language.

**Deterministic scan**: 4 findings across 2 files:
- PostoHighlightCard.tsx:29 — `border-l-4` side-tab accent (warning)
- Navbar.tsx:201, 211, 228 — `border-b-2` border accent on navigation items (3 warnings)

No browser overlay was injected (no live server running during this session).

## Overall Impression

Social-ASOF has a solid design foundation. The token system is well-built, dark mode is systematic, accessibility basics are in place, and the auth shell (login page) has genuine personality with its gradient, grid pattern, and glow effect. The diplomatic register comes through clearly.

The biggest opportunity is the Feed page, which suffers from decision overload — too many controls competing for attention simultaneously. Beyond that, the app feels functional but not yet delightful. There are no moments of personality outside the login screen, and power users have no accelerators.

## What's Working

1. **Token system and dark mode**: The CSS custom properties are comprehensive and the dark/light swap is systematic. No component needs dark-mode overrides when it uses tokens correctly — this is genuinely good infrastructure.
2. **Accessibility foundations**: 44px touch targets throughout, skip link, focus rings, aria labels, keyboard-navigable modals with focus trapping. These are not afterthoughts.
3. **Auth shell personality**: The login page's gradient, grid overlay, and glow orb create a distinctive first impression that no AI template would generate. This is the app at its most confident.

## Priority Issues

### [P1] Feed decision overload
**What**: The Feed page presents search input, category dropdown, 3 filter tabs, and the post list simultaneously — 7+ visible choices before the user sees any content.
**Why it matters**: Violates the "minimal choices" cognitive load principle. Users must process search + filter + category before engaging with content.
**Fix**: Progressive disclosure — combine search and category into a single expandable filter bar. Default to "Recentes" and hide filter tabs until the user taps a "Filter" trigger. Consider making category a pill selector that appears alongside search.
**Suggested command**: `$impeccable layout`

### [P1] PostoHighlightCard side-tab border (`border-l-4`)
**What**: A thick navy left border on the PostoHighlightCard — the #1 AI-UI tell flagged by the detector.
**Why it matters**: The design system explicitly states "border radius is deliberately minimal" and the entire app uses `0px` radius. A `border-l-4` accent stripe is the most recognizable AI-generation pattern. It contradicts the rectilinear brand identity.
**Fix**: Remove the left border. Use a subtle `bg-ice` or `bg-navy/5` background tint instead, or elevate the card with the existing shadow-sm token. The navy heading text already provides enough brand emphasis.
**Suggested command**: `$impeccable distill`

### [P1] Navbar shape inconsistency: rounded-full search + border-b-2 tabs
**What**: The Navbar uses `border-b-2` for active tab indicators (3 instances flagged by detector) and `rounded-full` for the search input, while every other input and card in the app uses `0px` radius (or no explicit radius).
**Why it matters**: The design system defines `border-radius: 0px` as default. The search input's pill shape and the bottom-border tab indicators both contradict this. They introduce two new shape languages that don't appear anywhere else.
**Fix**: Change the search input from `rounded-full` to the standard `0px` radius (or `rounded-sm` at most). Replace `border-b-2` active indicators with a background highlight (`bg-ice` or `bg-navy/5`) consistent with the rest of the app.
**Suggested command**: `$impeccable layout`

### [P2] No keyboard shortcuts or power-user accelerators
**What**: No keyboard shortcuts, no bulk actions, no way to navigate or act faster than clicking.
**Why it matters**: The ASOF community includes career diplomats who are likely proficient with keyboard-driven workflows. Without accelerators, the app feels slow for repeat users.
**Fix**: Add keyboard shortcuts for common actions (N for new post, / for search, Escape to close modals — already partially done). Show shortcut hints in tooltips.
**Suggested command**: `$impeccable delight`

### [P2] Empty states lack actionable guidance
**What**: Empty states like "Nenhum post encontrado" and "Nenhuma conversa" show an icon and message but no CTA. The Feed empty state says "Não encontramos publicações" but doesn't suggest creating a post or changing filters.
**Why it matters**: Empty states are a key onboarding moment. Without a next action, the user is stranded.
**Fix**: Add a primary CTA button to each empty state. "Nenhum post encontrado" → add a "Criar Publicação" button. "Nenhuma conversa" → add a "Buscar Membros" link.
**Suggested command**: `$impeccable onboard`

### [P2] PostEditor uses window.confirm for draft discard
**What**: Line 136 of PostEditor.tsx uses `window.confirm("Deseja realmente descartar o rascunho?")` instead of the ConfirmDialog component.
**Why it matters**: Known feedback item — window.confirm is inaccessible, blocks the main thread, and breaks the design system. The ConfirmDialog component exists and is used elsewhere.
**Fix**: Replace `window.confirm` with the existing `<ConfirmDialog>` component, matching the pattern already used for post deletion.
**Suggested command**: `$impeccable harden`

### [P3] Chat bubbles use the standard AI chat-bubble pattern
**What**: Messages.tsx uses `rounded-t-2xl rounded-l-2xl rounded-br-sm` for outgoing and `rounded-t-2xl rounded-r-2xl rounded-bl-sm` for incoming — the exact bubble pattern used by every AI chat interface.
**Why it matters**: While functional, it's visually generic. Since the app's design language is sharp-cornered, rounded bubbles feel inconsistent.
**Fix**: Either embrace rounded bubbles as a deliberate contrast (messaging has different affordances than cards), or use a subtler radius (`rounded-lg` at 8px) that bridges the sharp system with chat convention. Document the choice in DESIGN.md.
**Suggested command**: `$impeccable bolder`

### [P3] Toast z-index conflicts with logout dialog
**What**: Toasts use `z-[100]` while the logout dialog uses `z-50`. If a toast appears during logout confirmation, it renders on top of the dialog.
**Why it matters**: Z-index collisions create confusing layering where the user can't tell which surface is active.
**Fix**: Establish a semantic z-index scale in DESIGN.md (dropdown → sticky → modal-backdrop → modal → toast → tooltip). Apply it: modal dialogs should be z-50, toasts z-60, not z-100.
**Suggested command**: `$impeccable harden`

## Persona Red Flags

**Alex (Power User)**: No keyboard shortcuts anywhere. Can't filter by multiple categories. Can't bulk-delete or batch-edit posts. The Feed forces sequential scrolling with infinite load but no jump-to-date or search-within-results. Will feel slow after first week.

**Jordan (First-Timer)**: The Tour component exists but there's no obvious trigger for it. The PostEditor's category dropdown shows raw enum values like "VIDA_EXTERIOR" instead of "Vida no Exterior" (the translated labels are only in the option text, but the visual mapping is unclear). No empty-state guidance beyond "nothing here."

**Sam (Accessibility-Dependent)**: Skip link is present. Focus rings exist via `focus:ring-2 focus:ring-navy`. But the Navbar's active tab indicator uses `border-b-2` which is only visual — screen readers rely on `aria-current` (present, good). The chat messages use `role="log"` and `aria-live="polite"` correctly. Missing: no visible focus indicator on the search input's dropdown results, and the dropdown uses `mousedown` instead of `click` which may interfere with assistive tech.

## Minor Observations

- The Feed's "Você atingiu o fim do feed" text uses uppercase tracking-widest styling that's more suited to labels than body text
- Profile edit modal uses `bg-white/50` on inputs which reduces contrast below WCAG AA
- The `<select>` category dropdown in PostEditor has no custom styling beyond border/ring — it renders as a native OS dropdown which is functional but visually inconsistent
- Login form has `rounded-none` on inputs (consistent) but the search bar uses `rounded-full` (inconsistent)
- The `institutional-gold` color (`#c4a35a`) is defined in tokens but only appears in the Navbar bottom border shadow and auth shell — underutilized as a brand accent
- Dark mode search input placeholder uses `placeholder-slate/50` which may not meet contrast requirements
